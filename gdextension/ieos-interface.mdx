title: IEOS Singleton Interface
description: Low-level C++ singleton for direct Epic Online Services SDK access
---

The IEOS class is a C++ singleton that provides direct bindings to the Epic Online Services SDK. It's the lowest-level interface available in the plugin, offering one-to-one mappings to EOS SDK functions.

## Overview

IEOS is defined in `ieos.h` and `ieos.cpp` and serves as the bridge between Godot and the native EOS SDK. It:

- Extends `RefCounted` for Godot memory management
- Implements the singleton pattern for global access
- Exposes all EOS interface methods to GDScript
- Manages EOS SDK handles and state
- Emits signals for asynchronous callbacks

## Getting the Singleton

```gdscript
# Get the IEOS singleton
var ieos = IEOS.get_singleton()

# Or access via EOS wrapper
var ieos = EOS.get_instance()
```

## Core Methods

### Platform Management

<ParamField path="platform_interface_initialize(options)" type="int">
  Initializes the EOS SDK. Must be called before creating a platform instance.
  
  **Parameters:**
  - `options`: RefCounted containing initialization parameters
  
  **Returns:** EOS result code
</ParamField>

<ParamField path="platform_interface_create(options)" type="bool">
  Creates the EOS platform instance. Required for all EOS functionality.
  
  **Parameters:**
  - `options`: RefCounted containing platform configuration
  
  **Returns:** `true` if successful, `false` otherwise
</ParamField>

<ParamField path="platform_interface_release()" type="void">
  Releases the platform instance and associated resources.
</ParamField>

<ParamField path="platform_interface_shutdown()" type="int">
  Shuts down the EOS SDK. Should be called before application exit.
  
  **Returns:** EOS result code
</ParamField>

### Tick Method

<ParamField path="tick()" type="void" static>
  Processes EOS SDK callbacks and events. Must be called regularly (typically in `_process`).
  
  This is a static method that automatically operates on the singleton instance.
</ParamField>

```gdscript
func _process(_delta):
    IEOS.tick()
```

### Utility Methods

<ParamField path="is_operation_complete(result_code)" type="bool">
  Checks if an EOS operation has completed (successfully or with error).
  
  **Parameters:**
  - `result_code`: Integer EOS result code
  
  **Returns:** `true` if operation is complete
</ParamField>

## Platform Interface Methods

These methods manage platform-level functionality:

- `platform_interface_get_active_country_code(user_id)`: Get user's country code
- `platform_interface_get_active_locale_code(user_id)`: Get user's locale
- `platform_interface_get_application_status()`: Get application state
- `platform_interface_get_network_status()`: Get network connectivity state
- `platform_interface_set_application_status(status)`: Update application state
- `platform_interface_set_network_status(status)`: Update network state
- `platform_interface_check_for_launcher_and_restart()`: Check for Epic Games Launcher

## Service Interface Methods

IEOS exposes methods for all EOS services. Each follows the pattern:

```
{service}_interface_{method_name}(options)
```

### Authentication (Auth)

```gdscript
# Login
IEOS.auth_interface_login(options)

# Logout
IEOS.auth_interface_logout(options)

# Get login status
var status = IEOS.auth_interface_get_login_status(local_user_id)

# Get logged in accounts
var count = IEOS.auth_interface_get_logged_in_accounts_count()
var account_id = IEOS.auth_interface_get_logged_in_account_by_index(0)

# Copy tokens
var token_result = IEOS.auth_interface_copy_user_auth_token(options, local_user_id)
```

**Signals:**
- `auth_interface_login_callback(data: Dictionary)`
- `auth_interface_logout_callback(data: Dictionary)`
- `auth_interface_login_status_changed(data: Dictionary)`

## Signal Naming Convention

All IEOS signals follow the pattern:

```
{interface}_interface_{event}_callback
```

For example:
- `auth_interface_login_callback`
- `lobby_interface_create_lobby_callback`
- `stats_interface_query_stats_callback`

## Callback Data Structure

All callback signals receive a Dictionary with at minimum:

```gdscript
{
    "result_code": int,  # EOS result code
    # ... additional data specific to the callback
}
```

Common fields:
- `local_user_id`: The user who initiated the operation
- `client_data`: Custom data passed in the options
- Operation-specific data (e.g., `lobby_id`, `achievement_id`)

## Internal State

IEOS maintains internal handles for all EOS interfaces:

```cpp
EOS_HAchievements s_achievementsInterface;
EOS_HAuth s_authInterface;
EOS_HConnect s_connectInterface;
EOS_HFriends s_friendsInterface;
EOS_HLobby s_lobbyInterface;
EOS_HP2P s_p2pInterface;
EOS_HPlatform s_platformInterface;
// ... and many more
```

These are managed internally and not directly accessible from GDScript.

## Validation Flag

```gdscript
# Check if EOS platform is initialized
if IEOS.isEOSValid:
    # Safe to use EOS features
    pass
```

The `isEOSValid` property indicates whether the platform has been successfully created.

## Best Practices

### Use the EOS Wrapper

For most use cases, prefer the EOS wrapper classes over direct IEOS calls:

```gdscript
# Prefer this:
EOS.Auth.AuthInterface.login(options)

# Over this:
IEOS.auth_interface_login(options)
```

The wrapper provides:
- Type-safe option classes
- Better IDE autocomplete
- More idiomatic GDScript code

### Connect Signals Early

Connect to callback signals before making calls:

```gdscript
# Good
IEOS.connect("auth_interface_login_callback", _on_login)
IEOS.auth_interface_login(options)

# Bad - might miss the callback
IEOS.auth_interface_login(options)
IEOS.connect("auth_interface_login_callback", _on_login)
```

### Tick Regularly

Ensure `IEOS.tick()` is called every frame:

```gdscript
func _process(_delta):
    IEOS.tick()
```

### Check Validity

Verify platform is initialized before use:

```gdscript
if not IEOS.isEOSValid:
    push_error("EOS platform not initialized")
    return
```

## Comparison with EOS Wrapper

| Feature | IEOS | EOS Wrapper |
|---------|------|-------------|
| Type safety | Minimal | Strong |
| IDE support | Basic | Excellent |
| Ease of use | Low | High |
| Performance | Direct | Negligible overhead |
| Documentation | SDK reference | Built-in |
| Use case | Advanced/custom | Standard |
