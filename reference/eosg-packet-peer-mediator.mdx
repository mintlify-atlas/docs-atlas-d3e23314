## Overview

EOSGPacketPeerMediator is a singleton class that manages all active EOSGMultiplayerPeer instances. It receives packets from the EOS P2P interface every process frame, routes them to the appropriate multiplayer instance based on socket ID, and forwards EOS connection notifications.

**Extends:** `RefCounted`

**Source Files:**
- `eosg_packet_peer_mediator.h:50`
- `eosg_packet_peer_mediator.cpp:1`

## How It Works

1. **Registration**: Multiplayer instances register their socket ID when they become active
2. **Packet Routing**: Every process frame, the mediator polls packets from EOS and sorts them by destination socket
3. **Priority Queuing**: Peer ID packets are pushed to the front of the queue for faster connection establishment
4. **Notification Forwarding**: EOS connection callbacks (established, interrupted, closed, requests) are forwarded to the appropriate instance
5. **Connection Request Management**: Holds connection requests if no matching socket exists, forwards them when a matching socket is opened
6. **Unregistration**: Instances unregister when they close

## Packet Queuing Behavior

### Priority System

The mediator uses a priority queuing system:

- **Peer ID packets** (EVENT_RECIEVE_PEER_ID): Pushed to front of queue
- **Regular packets** (EVENT_STORE_PACKET): Pushed to back of queue

This ensures connection handshakes complete quickly.

**Source:** `eosg_packet_peer_mediator.cpp:109`

### Queue Size Management

When the queue reaches the size limit:

1. The `packet_queue_full` signal is emitted
2. Packet polling stops until packets are consumed
3. New packets from EOS are not polled (they remain in the EOS queue)

**Source:** `eosg_packet_peer_mediator.cpp:57`

## Usage Examples

### Monitoring Packet Queue

```gdscript
extends Node

var mediator: EOSGPacketPeerMediator

func _ready():
    mediator = EOSGPacketPeerMediator.get_singleton()
    mediator.packet_queue_full.connect(_on_queue_full)

func _process(_delta):
    # Monitor queue usage
    var total_packets = mediator.get_total_packet_count()
    var queue_limit = mediator.get_queue_size_limit()
    var usage_percent = (total_packets / float(queue_limit)) * 100
    
    if usage_percent > 80:
        print("WARNING: Queue at %.1f%% capacity" % usage_percent)

func _on_queue_full():
    print("CRITICAL: Packet queue full!")
    # Option 1: Increase the limit
    mediator.set_queue_size_limit(mediator.get_queue_size_limit() * 2)
    
    # Option 2: Log which sockets are using the most space
    var sockets = mediator.get_sockets()
    for socket_id in sockets:
        var count = mediator.get_packet_count_for_socket(socket_id)
        print("Socket '%s': %d packets" % [socket_id, count])
```

### Tracking Connection Requests

```gdscript
extends Node

var pending_requests: Array = []

func _ready():
    var mediator = EOSGPacketPeerMediator.get_singleton()
    mediator.connection_request_received.connect(_on_request_received)
    mediator.connection_request_removed.connect(_on_request_removed)

func _on_request_received(data: Dictionary):
    pending_requests.append(data)
    print("Connection request received for socket: ", data["socket_id"])
    print("From user: ", data["remote_user_id"])
    print("Total pending: ", mediator.get_connection_request_count())

func _on_request_removed(data: Dictionary):
    pending_requests.erase(data)
    print("Connection request removed: ", data["remote_user_id"])
```

## Internal Behavior

### Process Frame Callback

Every frame, the mediator:

1. Checks if a local user ID is set (skips if not logged in)
2. Checks if any sockets are registered (skips if none)
3. Checks if queue is full (stops polling if full)
4. Polls packets from EOS P2P interface
5. Routes packets to appropriate socket queues
6. Prioritizes peer ID packets by pushing to front

**Source:** `eosg_packet_peer_mediator.cpp:52`

### Connection Request Handling

When a connection request is received:

1. Check if a matching socket is registered
2. If yes: Forward immediately to that multiplayer instance
3. If no: Store in pending list and emit `connection_request_received`
4. When a new socket registers: Forward any matching pending requests
5. When a connection closes: Remove associated pending requests

**Source:** `eosg_packet_peer_mediator.cpp:363`

### Peer Registration

When an EOSGMultiplayerPeer registers:

1. Validates the peer has a valid socket
2. Checks the socket isn't already registered
3. Creates a new packet queue for the socket
4. Forwards any pending connection requests for that socket

**Source:** `eosg_packet_peer_mediator.cpp:149`

### Peer Unregistration

When an EOSGMultiplayerPeer unregisters:

1. Clears all queued packets for that socket
2. Removes the socket from the active list
3. Removes the packet queue

**Source:** `eosg_packet_peer_mediator.cpp:170`

## Performance Considerations

### Queue Size Tuning

- **Default (5000)**: Suitable for most games with moderate packet rates
- **High traffic (10000+)**: Lobby servers, MMOs, or games with many peers
- **Low traffic (1000-3000)**: Small games, turn-based games

**Source:** `eosg_packet_peer_mediator.h:62`

### Memory Usage

Each queued packet includes:
- Packet data (up to ~1170 bytes)
- Sender information
- Channel information
- Socket routing information

At the default limit of 5000 packets, worst-case memory usage is approximately 6-7 MB.

### Processing Overhead

The mediator processes packets on the main thread during `process_frame`. If packet processing becomes a bottleneck:

1. Reduce queue size to force packet consumption
2. Optimize packet handling in your multiplayer code
3. Reduce packet send frequency
4. Use larger packets with batched data instead of many small packets

## Important Notes

- The mediator is a singleton - only one instance exists per game
- It automatically initializes on Connect interface login
- Packets are processed every frame on the main thread
- Peer ID packets are prioritized for faster connection handshakes
- Connection requests without matching sockets are held indefinitely (until timeout or socket opens)
- The queue size limit applies to ALL sockets combined, not per socket
- When the queue is full, new packets remain in the EOS queue and are polled when space is available
